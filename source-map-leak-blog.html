From Source Maps to Source Leaks: What Anthropic's Claude Code Incident Teaches Every Developer

A single misconfigured build step just exposed 512,000 lines of proprietary code. Here's how source maps work, why they're dangerous, and what you can do about it.

March 31, 2026 · 10 min read

What Just Happened

Earlier today, security researcher Chaofan Shou discovered something extraordinary: Anthropic had accidentally shipped a .map file inside version 2.1.88 of their @anthropic-ai/claude-code npm package. That single file contained everything needed to reconstruct the entire original TypeScript source code — nearly 1,900 files, over 512,000 lines — of one of the most commercially significant AI developer tools in existence.

Within hours, the code was mirrored to GitHub, forked over 41,500 times, and dissected across Hacker News, Twitter, and every security blog with a pulse. Anthropic pulled the affected version, but the damage was done. The source is now permanently in the wild.

The irony? This is Anthropic's third known source map leak. Versions 0.2.8 and 0.2.28 in 2025 had the same problem. And buried in the leaked code is an entire subsystem called "Undercover Mode" specifically designed to prevent internal information from leaking through git commits. They built a system to stop leaks — and then leaked the system.

Source Maps: The Innocent Debugging Tool

To understand why this is such a big deal, you need to understand what source maps are and why every JavaScript project generates them.

Modern JavaScript and TypeScript code goes through extensive transformation before it reaches production. TypeScript gets compiled to JavaScript. Modules get bundled into single files. Code gets minified — variable names shortened, whitespace stripped, logic compressed. The result is fast and small, but completely unreadable.

Source maps exist to bridge that gap. They're JSON files that create a precise, character-by-character mapping between the transformed output and the original source code. When something crashes in production, your browser's DevTools uses the source map to show you the error in your original, readable code — not in line 1, column 48,293 of a minified bundle.

The Anatomy of a .map File

JSON 
{
  "version": 3,
  "file": "app.min.js",
  "sources": ["src/auth/oauth.ts", "src/agent/main-loop.ts"],
  "sourcesContent": ["// full original source code here..."],
  "names": ["handleAuth", "sessionToken", "agentLoop"],
  "mappings": "AAAA,SAAS,GAAG;AACZ,..."
}

Each field plays a specific role. sources lists every original file path. names preserves the original variable and function names. mappings encodes the position-to-position correspondence using a compact Base64 VLQ encoding — a variable-length scheme where small numbers (common in incremental offsets) use just one character, keeping the file reasonably sized.

And then there's sourcesContent — the field that makes this a nuclear-grade leak vector. Many build tools populate it by default, embedding the complete, verbatim original source code directly into the JSON. If this field is present, you don't need access to the source repository. You don't need to reverse-engineer anything. You just read the file.

TypeScript Source → Compiler + Bundler → Minified JS + .map file
The source map is a byproduct of the build process — and it contains everything needed to reverse the transformation.

How the Browser Uses It

The connection between a minified file and its source map is established by a single comment at the end of the generated JavaScript:

JS 
//# sourceMappingURL=app.min.js.map

When browser DevTools detect this directive, they fetch the .map file, decode the mappings string, and build a bidirectional lookup table. Every position in the minified code gets linked to the corresponding position in the original source. Breakpoints, stack traces, and error messages all resolve to the original file names, line numbers, and variable names.

Critically, this only happens when DevTools are open. Source maps have zero runtime performance impact — they're a pure debugging artifact. Which is exactly why they're so easy to forget about at deploy time.

Why Leaking a Source Map Is Leaking Your Source Code

This isn't a theoretical risk. Here's what a source map exposes in practice:

  • Complete original source — If sourcesContent is populated, every line of every original file is embedded in the JSON. No reverse engineering required.
  • Project architecture — The sources array reveals your entire directory structure: src/agent/main-loop.ts, src/tools/bash/executor.ts, src/permissions/trust-model.ts. An attacker immediately understands how your system is organized.
  • Original identifiers — The names array and the mapping data restore every variable, function, and class name that the minifier stripped away. H15 becomes createRequire; iS6 becomes defineProperty.
  • Internal comments and TODOs — Developer notes, hack warnings, performance annotations, and internal links that were never meant to be public.
  • Business logic — Pricing calculations, permission models, API integration details, feature flag configurations.

What Was Found in the Claude Code Leak

The community wasted no time analyzing the exposed code. Among the discoveries:

  • Internal model codenames — "Capybara" for Claude 4.6, "Fennec" for Opus 4.6, and an unreleased "Numbat" still in testing.
  • Anti-distillation defenses — A flag called ANTI_DISTILLATION_CC that injects fake tool definitions into API requests to poison any competitor trying to train on recorded Claude Code traffic.
  • KAIROS autonomous mode — A background daemon feature mentioned 150+ times in the source, including an autoDream process that performs "memory consolidation" while the user is idle.
  • 44 hidden feature flags — Covering features that are fully built but gated behind compile-time flags, essentially revealing Anthropic's unshipped product roadmap.
  • Performance metrics — Internal comments noting a 29–30% false claims rate in Capybara v8, a regression from 16.7% in v4.

Supply-chain alert: A separate, concurrent attack on the axios npm package occurred hours before the leak. If you installed Claude Code via npm on March 31, 2026, between 00:21 and 03:29 UTC, check your lockfiles for axios versions 1.14.1 or 0.30.4. If found, treat the machine as compromised and rotate all secrets.

How This Keeps Happening

The root cause is almost always the same: a gap between what the build tool generates and what the publish configuration includes.

Most bundlers — Webpack, esbuild, Vite, and notably Bun (which Claude Code uses) — generate source maps by default or with minimal configuration. The .map files end up in the build output directory alongside the minified JavaScript.

When you run npm publish, npm packages everything in the directory unless explicitly told otherwise. If your .npmignore doesn't exclude *.map files, or if your package.json "files" field is too broad, the source maps ship to the registry.

In Anthropic's case, there's an additional twist: Bun has an open bug (oven-sh/bun#28001, filed March 11) reporting that source maps are served in production mode even when they shouldn't be. Anthropic acquired Bun late last year. Their own toolchain may have exposed their own product.

How to Protect Yourself

1. Configure Your Build Tool

Explicitly disable source map generation for production builds, or use a "hidden" mode that generates the file without embedding the sourceMappingURL directive.

webpack.config.js 
// Generate .map files but don't reference them in the bundle
module.exports = {
  devtool: "hidden-source-map",
};
tsconfig.json 
{
  "compilerOptions": {
    "sourceMap": false  // or use declarationMap only
  }
}

2. Exclude from Publishing

Add explicit exclusion rules to your npm package configuration:

.npmignore 
*.map
*.js.map
*.css.map

Or use the allowlist approach in package.json, which is safer because it only includes what you explicitly specify:

package.json 
{
  "files": ["dist/**/*.js", "dist/**/*.d.ts", "README.md"]
}

3. Block at the Web Server

For frontend deployments, add a server-level safeguard:

nginx.conf 
# Return 404 for any source map request
location ~* \.map$ {
    return 404;
}

4. Validate in CI

Add a check to your CI pipeline that fails the build if any .map files are present in the publish artifact:

Shell 
# Fail if any .map files exist in the package
npm pack --dry-run 2>&1 | grep -q '\.map' && \
  echo "ERROR: Source maps found in package!" && exit 1

5. Use Source Maps Only for Error Monitoring

Upload .map files to your error tracking service (Sentry, Datadog, Bugsnag) but never deploy them to public-facing servers. These services accept private source map uploads and use them server-side to symbolicate stack traces — your users and attackers never see them.

The Bigger Picture

Source maps are a solved problem — the tooling to exclude them exists, the best practices are well-documented, and the risks are widely known. Yet this keeps happening, even to companies with billions in revenue and world-class engineering teams.

The lesson isn't that source maps are bad. They're an essential debugging tool. The lesson is that every build artifact is a potential leak vector, and the gap between "development convenience" and "production security" is exactly one forgotten configuration line.

For any team shipping JavaScript or TypeScript — whether it's a SaaS platform, a CLI tool, or a browser extension — the checklist is simple: generate source maps for development, upload them to your error tracker, and make absolutely certain they never reach your users.

Anthropic will recover. The code can be refactored, the feature flags renamed, the internal codenames rotated. But the strategic surprise — 44 unshipped features, an unreleased autonomous agent mode, internal performance benchmarks — that can never be un-leaked.

Check your .npmignore today. It takes thirty seconds. The alternative takes considerably longer to clean up.
